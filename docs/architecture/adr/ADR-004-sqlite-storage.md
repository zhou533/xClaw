# ADR-004: 选用 SQLite 作为数据存储

## 背景

xClaw 需要持久化以下数据：
- 对话历史（消息记录、工具调用日志）
- 会话元数据（通道来源、时间戳）
- 长期记忆（向量嵌入可选）
- 用户配置（运行时覆盖项）

作为单用户个人助手，数据量和并发需求有限，但需要跨平台零依赖部署。

## 决策

选用 **SQLite**（通过 `rusqlite` crate）作为主数据存储。

SQLite 是嵌入式关系数据库，直接编译进应用二进制文件，无需独立数据库进程。
启用 WAL（Write-Ahead Logging）模式以支持并发读写。

辅以 SQLite FTS5 扩展提供全文搜索能力，满足对话历史检索需求。

## 影响

### 正面影响
- **零依赖**：嵌入式部署，无需安装和配置数据库服务
- **跨平台**：macOS / Windows / Linux 完全一致的行为
- **数据便携**：单个 `.db` 文件，易于备份、迁移和调试
- **性能优异**：单用户场景下读写性能远超需求（~100K 次写入/秒）
- **FTS5**：内置全文搜索，满足消息检索需求
- **成熟稳定**：经过 20+ 年验证的嵌入式数据库

### 负面影响
- **并发写入限制**：WAL 模式下支持并发读但写入仍然序列化（单用户场景影响可忽略）
- **无内置向量搜索**：如需语义检索需外挂方案（sqlite-vss 或嵌入式 qdrant）
- **无网络访问**：多设备同步需要额外同步层（如 Litestream）

### 备选方案
- **PostgreSQL**：功能强大，但需独立进程，部署复杂度大幅增加，单用户场景过于重量级
- **RocksDB**：嵌入式 KV 存储，但缺乏关系查询能力，对话历史查询不便
- **Sled**：纯 Rust 嵌入式数据库，但成熟度和稳定性不及 SQLite
- **纯文件系统（JSON/JSONL）**：最简单，但查询能力弱，数据量增长后性能下降
